# FitFindr

FitFindr is a multi-step orchestration agent that coordinates three distinct tools to search for secondhand clothing, generate tailored styling advice based on a user's existing wardrobe, and draft a shareable social media caption. 

## Tool Inventory

### 1. `search_listings`
* **Purpose:** Searches the mock dataset for secondhand items matching user criteria.
* **Inputs:** * `description` (str): Keywords describing the desired item.
  * `size` (str | None): Optional size filter.
  * `max_price` (float | None): Optional maximum price ceiling.
* **Outputs:** `list[dict]` - A list of dictionaries representing matching listings sorted by relevance.

### 2. `suggest_outfit`
* **Purpose:** Uses the Groq LLM to suggest 1-2 complete outfit combinations using a newly thrifted item and the user's existing wardrobe.
* **Inputs:**
  * `new_item` (dict): The listing dictionary of the item selected from the search.
  * `wardrobe` (dict): A dictionary containing an `items` list of the user's existing clothing.
* **Outputs:** `str` - A paragraph containing the styling advice.

### 3. `create_fit_card`
* **Purpose:** Uses the Groq LLM to generate a short, authentic social media caption (OOTD style) based on the outfit suggestion.
* **Inputs:**
  * `outfit` (str): The outfit suggestion generated by `suggest_outfit`.
  * `new_item` (dict): The listing dictionary for the thrifted item.
* **Outputs:** `str` - The generated 2-4 sentence caption.

## Planning Loop
The agent executes an adaptive conditional loop. After parsing the user query, it calls `search_listings`. The agent then checks the state of the return value. **If** the list is empty `[]`, it assigns a user-friendly error message to `session["error"]` and terminates the execution immediately, preventing downstream LLM tools from receiving empty data. **Else**, it extracts the top item from the results, assigns it to the session state, and passes it into `suggest_outfit` and `create_fit_card` sequentially. 

## State Management
Information flows between tools via a central `session` dictionary that acts as the single source of truth for the run. For example, the agent saves the result of the search tool into `session["selected_item"]`. Rather than requiring the user to confirm or re-enter this item, the agent passes `session["selected_item"]` directly as the `new_item` argument into the `suggest_outfit` tool. The output of that tool is saved to `session["outfit_suggestion"]`, which is piped directly into `create_fit_card`.

## Error Handling
Every tool handles its own failure state gracefully without raising Python exceptions:
* **`search_listings`:** If a user searches for an impossible item (e.g., "$5 designer ballgown"), the tool safely returns `[]`. The agent detects this and sets an error message: *"Couldn't find anything matching that. Try loosening your size or price constraints!"*
* **`suggest_outfit`:** If the user has an empty wardrobe (tested using `get_empty_wardrobe()`), the tool catches the empty dictionary key and dynamically swaps the system prompt to provide general styling advice instead of crashing.
* **`create_fit_card`:** A guard clause `if not outfit:` catches empty inputs and safely returns an error string: *"Could not generate fit card: outfit details are missing."*

## Spec Reflection
* **How the spec helped:** Explicitly writing out the "Empty Wardrobe" failure mode in `planning.md` made the implementation of `suggest_outfit` incredibly smooth. Because I knew exactly what state to look for, I could easily write the `if not wardrobe.get("items"):` branch to trigger a fallback prompt.
* **How implementation diverged:** Initially, the plan was a bit vague on how the agent would extract the `size` and `max_price` parameters to pass to the search tool. In the actual implementation of `agent.py`, I chose to use Python `re` (regex) to parse the size and price strings out of the query instead of using an LLM call, as regex was faster and perfectly reliable for standard inputs.
